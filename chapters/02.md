# Verifiers

Now that we understand how LLMs act as agents within an environment and how rewards guide their learning, let's see it in action.

To develop our Tic Tac Toe Environment, we'll use [Verifiers](https://github.com/PrimeIntellect-ai/verifiers), an open-source library maintained by William Brown and the Prime Intellect team.

Verifiers provides modular components to create RL environments for LLM agents, which can then be used for both evaluation and training.

## Why use a library at all? Why not just code everything from scratch?
It's a matter of simplicity and convenience.

Verifiers handles much of the boilerplate that comes up when working with RL environments for LLMs.

- Environments are Python packages that can be easily installed and distributed.

- The library provides base classes for various setups: single-turn Environments with just one interaction between the model and the env, multi-turn Environments, Tool Environments - where the model is equipped with tools, and several others.

- It includes abstractions for parsing model responses and defining reward functions.

- Verifiers abstracts model serving: it expects an OpenAI-compatible API endpoint, allowing you to use OpenAI models, OpenRouter, or local models served via vLLM.

- * It handles async interaction and parallel trajectories, so you focus on environment logic.

- For training, Verifiers comes with its own trainer and integrates with other frameworks such as PRIME-RL, Tinker, and SkyRL.

In short, Verifiers lets us focus on the task and the rewards rather than the infrastructure.

Verifiers is also tightly integrated with the [Environments Hub](https://app.primeintellect.ai/dashboard/environments): a community hub for discovering and sharing environments.

If you want to explore the library further, check out the [documentation](https://docs.primeintellect.ai/verifiers).

It's worth mentioning that while we'll use Verifiers in this course, there are also other efforts to simplify and standardize
RL environments for LLMs, like [OpenEnv](https://github.com/meta-pytorch/OpenEnv) and [Atropos](https://github.com/NousResearch/atropos).

## Single Turn Environment

Before tackling multi-turn problems, let's start simple.

We'll pick an illustrative example from Verifiers: [Reverse Text Environment](https://github.com/PrimeIntellect-ai/verifiers/tree/main/environments/reverse_text), a simple environment to evaluate or train Language Models on their ability to reverse text.

Here's the code:

```python
from datasets import load_dataset

import verifiers as vf


def load_environment(
    dataset_name: str = "PrimeIntellect/Reverse-Text-RL",
    dataset_split: str = "train",
    system_prompt: str
    | None = "Reverse the text character-by-character. Put your answer in <reversed_text> tags.",
) -> vf.Environment:
    train_dataset = load_dataset(dataset_name, split=dataset_split).map(
        lambda x: {
            "question": x["prompt"],
            "answer": x["prompt"][::-1],
            "info": {},
            "task": "reverse-text",
        }
    )
    train_dataset = train_dataset.remove_columns(["prompt"])

    parser = vf.XMLParser(["reversed_text"], answer_field="reversed_text")

    def lcs_reward_func(completion, answer, **kwargs) -> float:
        """
        LCS ratio of the reversed prompt and the parsed completion.
        """

        def lcs_ratio(x: str, y: str) -> float:
            """
            Return the longest common subsequence ratio of x and y.
            """
            from difflib import SequenceMatcher

            return SequenceMatcher(None, x, y).ratio()

        response = parser.parse_answer(completion) or ""
        return lcs_ratio(response, answer)

    rubric = vf.Rubric(
        funcs=[
            lcs_reward_func,
        ],
        weights=[1.0],
    )

    vf_env = vf.SingleTurnEnv(
        dataset=train_dataset,
        system_prompt=system_prompt,
        parser=parser,
        rubric=rubric,
    )
    return vf_env
```

What's going on here?

`load_environment` is the entry point for every Verifiers environment. It contains all the setup logic needed for both evaluation and training.

First, a dataset is loaded and mapped. The [default dataset](https://huggingface.co/datasets/PrimeIntellect/Reverse-Text-RL) contains 1000 text paragraphs stored in the `prompt` column.

During the mapping step, we transform the original dataset: `question` will contain the text paragraph, while `answer` will simply be the reversed text.

Next, an `XMLParser` is initialized. It extracts the text within the `<reversed_text>` tags, as specified in the system prompt.

We then define `lcs_reward_func`, a reward function that compares the model's parsed answer to the ground truth and returns the Longest Common Subsequence ratio.

A `Rubric`, a collection of weighted reward functions, is defined next. In this case, it contains only our LCS function with a weight of 1.0.

Finally, we initialize a `SingleTurnEnv`, passing in all the components defined above.

*How does this come to life?*

When we run evaluation, for example using `vf-eval reverse-text -m gpt-4.1-mini -n 5 -r 3`, the following happens:
- `load_environment` is invoked
- 5 examples are taken from the dataset, and each example is used 3 times (3 rollouts): each rollout gets the same question but may produce different completions due to model randomness. We'll get 15 rollouts in total.
- For each rollout:
  - a conversation is prepared, consisting of the system prompt and a user message containing the text to reverse (`question`)
  - the conversation is sent to the model (`gpt-4.1-mini`)
  - the model generates a response (e.g: "I'll reverse the text 'hello' character by character. <reversed_text>olleh</reversed_text>")
  - the response is parsed, extracting the answer "olleh"
  - the reward `lcs_reward_func` is computed and results are saved
- At the end of evaluation, we get summary statistics for rewards (averages and standard deviations), plus per-rollout breakdowns grouped by example.

Training follows the same core mechanism, with the additional step of updating model parameters. We'll look at that in more detail later.

## Multi Turn Environment

I promise we'll dive into Tic Tac Toe soon, but to keep learning gradually, let's look at another  example from Verifiers: [the DoubleCheck environment](https://github.com/PrimeIntellect-ai/verifiers/tree/main/environments/doublecheck). Here, the model answers a math question and is then asked "Are you sure?".

This is a multi-turn environment, similar in spirit to Tic Tac Toe, in which each trajectory involves multiple interactions between the model and the environment.

Let's take a look at the code:

```python
import verifiers as vf
from verifiers.types import Messages, State
from verifiers.rubrics.math_rubric import MathRubric
from verifiers.utils.data_utils import load_example_dataset

SIMPLE_PROMPT = """
Respond in the following format, using careful step-by-step reasoning.

<reasoning>
...
</reasoning>
<answer>
...
</answer>
"""


class DoubleCheckEnv(vf.MultiTurnEnv):
    def __init__(self, **kwargs):
        super().__init__(**kwargs)

    @vf.stop
    async def double_checked(self, state: State) -> bool:
        return len(state["trajectory"]) == 2

    async def env_response(
        self, messages: Messages, state: State, **kwargs
    ) -> Messages:
        """Generate a response from the environment."""
        return [{"role": "user", "content": "Are you sure?"}]


def load_environment(
    dataset_name: str = "math",
    dataset_split: str = "train",
    num_train_examples: int = -1,
):
    dataset = load_example_dataset(dataset_name, dataset_split, n=num_train_examples)
    rubric = MathRubric()
    vf_env = DoubleCheckEnv(
        dataset=dataset,
        system_prompt=SIMPLE_PROMPT,
        few_shot=[],
        parser=rubric.parser,
        rubric=rubric,
    )
    return vf_env
```

Let's examine the main differences from the Single Turn environment seen before, focusing on what `MultiTurnEnv` introduces.

- [`State`](https://docs.primeintellect.ai/verifiers/source/api_reference#state-dictionary) makes its first appearance here, as a dictionary that tracks information during a rollout. Although not used here, we can set the initial state through the `setup_state` method.

- `env_response`: Instead of the interaction ending after one turn, the environment can reply with a list of messages. Here, the model statically returns "Are you sure?". In more complex environments, the response can be dynamically generated based on the current `State`.

- The `@vf.stop` decorator marks a method as a stopping condition. This method runs at every turn of the agent–environment interaction; once it returns `True`, the rollout terminates.

Under the hood, Verifiers runs a loop in which the model and the environment take turns exchanging messages and updating shared state until a stopping condition is met and the full trajectory can be evaluated.

## Other Environments

It's worth noting that Verifiers also offer other classes to build environments that address common use cases.
For example, `ToolEnv` is a multi-turn environment that includes tool calling. `MCPEnv` automatically connects to MCP servers and exposes their tools to the model.
To discover more about available environment patterns, check out [Verifiers docs](https://docs.primeintellect.ai/verifiers/environments).
